# T016 — Result tab + shared result surface

**Status:** planned  
**Maps to:** US-14 (dedicated result view), US-13 (clean Search), US-10 (reopen — carried over)  
**Depends on:** [T015 — URL input + Search relayout](T015.md)

## Goal

Move results off Search into their own dedicated **Result** tab. Introduce a shared result surface (one `ResultView` host) fed by a small controller: the Result tab is a permanent destination that shows an empty state until there's something to show; a successful identification publishes its result and the app auto-switches to the tab, and Search resets to a clean state. Reopening a past identification from History publishes into the same Result view (still with **no** API call), retiring the separate `HistoryDetailScreen`.

## Approach

Add a `ResultController` (`ChangeNotifier`) that holds the currently displayed result — image (bytes/url), `predictions`, and whether it's a saved reopen (`saved` + `savedAt`) — or `null` when there's nothing to show. `AppShell` gains a fourth **permanent** destination, Result, in fixed order **Search / Result / History / Settings** (static indices — no dynamic destination juggling). `ResultScreen` listens to the controller and renders either the empty state (`current == null`) or the `ResultView`. Construct the controller in `main.dart` (app-lifetime, like the stores) and thread it to `AppShell`.

`HomeScreen` gains an `onResult` hook (or the shared `ResultController` + a "switch to Result" callback). On a successful `_identify`, it: (1) auto-saves to history as today, (2) publishes the result via `ResultController.show(...)`, (3) asks `AppShell` to select the Result tab, and (4) clears its own inputs/state so Search is clean when revisited. `HistoryScreen` tiles, instead of pushing `HistoryDetailScreen`, call `ResultController.show(... saved: true, savedAt: entry.createdAt)` (reconstructed from the stored entry, no `ApiClient`) and switch to the Result tab.

```
ui/lib/
  services/result_controller.dart   # NEW — ChangeNotifier holding the active result
  screens/result_screen.dart        # NEW — Result tab; empty state OR ResultView from the controller
  app_shell.dart                     # permanent Search/Result/History/Settings destinations + selected index + auto-switch
  screens/home_screen.dart           # on success: show result + switch tab + reset self
  screens/history_screen.dart        # reopen → controller.show(saved) + switch tab (no detail push)
  screens/history_detail_screen.dart # REMOVED (replaced by the Result tab)
  main.dart                          # construct + thread ResultController into AppShell
```

### Proposed files

| File | Purpose |
|------|---------|
| `ui/lib/services/result_controller.dart` | `ResultController extends ChangeNotifier`; `ActiveResult? get current`, `bool get hasResult`, `void show(...)`, `void clear()` |
| `ui/lib/screens/result_screen.dart` | Scaffold (AppBar "Result"); `ListenableBuilder` on the controller → empty state when `current == null`, else `ResultView` (with saved badge for reopened entries) |
| `ui/lib/app_shell.dart` | Add the permanent Result destination (order: Search/Result/History/Settings); auto-switch to it on publish |
| `ui/lib/screens/home_screen.dart` | On success: `controller.show(...)` + request Result tab + reset `_bytes`/`_filename`/`_previewUrl`/`_predictions`/`_error` |
| `ui/lib/screens/history_screen.dart` | Tile `onTap` → `controller.show(saved:true, savedAt: entry.createdAt)` + switch to Result; drop `HistoryDetailScreen` import/push |

### Result tab behavior (US-14)

- **Permanent tab + empty state:** the Result destination is always present (`Key('nav-result')`, e.g. `Icons.insights` / `Icons.fact_check_outlined`). Before any result, `ResultScreen` shows a clear, flat empty state — e.g. an icon + "No result yet — identify a car from Search" (`Key('result-empty')`), consistent with the History empty state (`HistoryEmpty`).
- **Static indices:** destinations are fixed in order Search (0) / Result (1) / History (2) / Settings (3); the `IndexedStack` children match. No dynamic add/remove. Search stays the default landing section; preserve `Key('nav-search')` / `Key('nav-history')` / `Key('nav-settings')`.
- **Auto-switch on success:** publishing a fresh result selects the Result tab automatically (replacing its empty state with the result).
- **Search clean-reset:** immediately after a successful identify, `HomeScreen` clears its inputs and local result state so returning to Search shows the empty state (US-13). `IndexedStack` keeps `HomeScreen` alive, so this is an explicit `setState` reset.
- **Reopen path (US-10):** History reopen reconstructs the image (decode data URL for uploads, pass remote URL otherwise) + `predictions` purely from the stored `HistoryEntry` and publishes with `saved: true`. The Result tab must **not** construct or call `ApiClient`. Reuse the existing `ResultView` `saved`/`savedAt` badge (`Key('saved-result')`). Move the data-URL decode helper out of the deleted detail screen into a shared spot (e.g. on `HistoryEntry` or the result screen).
- **Fresh vs saved:** a fresh result clears the saved badge; the shared `ResultView` already supports both via its `saved` flag.

### Selectors for automation (T017)

- `Key('nav-result')` on the destination; `Key('result-screen')` on the tab body; `Key('result-empty')` on the empty state. Existing `Key('saved-result')`, `Key('top-prediction')`, and `PredictionList` semantics carry over via `ResultView`.

## Acceptance criteria

- [ ] A permanent Result tab is always present and shows a clear empty state before any result exists (US-14)
- [ ] A successful identification auto-switches to the Result tab and shows the full image + top-k (US-03/US-14)
- [ ] After identifying, Search is reset to a clean/empty state (US-13)
- [ ] Reopening a History entry shows it in the Result view, marked as saved, with **no** API call (US-10/US-14)
- [ ] `HistoryDetailScreen` is removed; both fresh and reopened results share one `ResultView` host
- [ ] Destinations are fixed (Search/Result/History/Settings) with stable selectors; Search remains the default landing section
- [ ] `flutter analyze` reports no new warnings

## Out of scope

- Re-running inference / "refresh" on a result (still out per backlog)
- Hiding the Result tab when empty, or requiring a manual switch (we chose a permanent tab + auto-switch)
- Editing results, sharing/exporting
- Playwright updates for the new flow (T017)
