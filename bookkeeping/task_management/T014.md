# T014 ŌĆö Image selection card

**Status:** planned  
**Maps to:** US-13 (focused Search screen)  
**Depends on:** nothing (can start in parallel with the theme track)

## Goal

Replace the "Upload image" button + separate inline preview with a single, self-contained `ImageSelectionCard`: one clickable card that *is* both the picker and the preview. Tapping anywhere on the card opens the file picker; once an image is chosen it previews inside the same card, which keeps a fixed footprint and offers a way to expand/inspect the image.

## Approach

Extract image selection into a reusable, mostly-presentational widget that takes the currently selected image (bytes and/or URL) and callbacks for "pick" and "expand". `HomeScreen` still owns the picking logic and selected-image state (T015 wires it in); this task delivers the card widget and its empty / filled / expand behaviors. The card replaces both the `OutlinedButton.icon('Upload image')` and the standalone `ImagePreview` on Search.

### Card behavior (US-13)

- **Fixed footprint:** the card is the same size whether empty or filled (e.g. fixed height ~220 matching today's preview, full available width). Selecting an image must **not** resize it; the image fits *within* the card (`BoxFit.contain`).
- **Single clickable surface:** the whole card is tappable (`InkWell`/`Material`) and triggers `onPick` ŌĆö there is no separate upload button. Disabled while `enabled == false` (e.g. during inference).
- **Empty state:** a clear prompt inside the card ŌĆö icon + text like "Tap to add a car photo" (can reuse the `directions_car_outlined` motif from `ImagePreview`).
- **Filled state:** the picked image (bytes) or network image (url) previewed inside; tapping still re-opens the picker to swap the image.
- **Symmetric padding:** the content inside the card has equal padding on all four sides (top/bottom == left/right) ŌĆö use a single `DriftSpacing` value, not the asymmetric layout today.
- **Expand/inspect:** an explicit affordance (e.g. a small `Key('expand-image')` icon button overlaid in a corner, shown only when an image is present) opens the image larger for inspection ŌĆö a simple full-bleed dialog/`InteractiveViewer` is sufficient. This control is separate from the tap-to-pick surface so the two don't conflict.
- **Theme-aware:** derive colors from `Theme.of(context).colorScheme` (works in light + dark, T012); keep it flat ŌĆö solid fill, `kDriftRadius`, no shadows/gradients.

### Selectors for automation (T017)

- Card root: `Key('image-card')` with a `Semantics` label (e.g. "Add a car photo"). The expand control: `Key('expand-image')`. Note: the old `Key('upload-button')` is **removed** ŌĆö T017 updates any test that targeted it to use `Key('image-card')`.

## Acceptance criteria

- [ ] Image selection is a single clickable card (no separate upload button); tapping it opens the picker (US-13)
- [ ] The card keeps a fixed footprint empty vs. filled; the chosen image previews inside without resizing the card (US-13)
- [ ] Padding inside the card is symmetric (top/bottom equal to left/right) (US-13)
- [ ] An expand/inspect affordance shows the image larger when one is selected (US-13)
- [ ] The card is theme-aware (legible in light + dark) and flat per `AGENTS.md`; labeled + reachable by Playwright (US-13)
- [ ] `flutter analyze` reports no new warnings
