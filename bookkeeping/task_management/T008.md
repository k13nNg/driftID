# T008 — Reopen a past result

**Status:** planned  
**Maps to:** US-10 (reopen a past result in full)  
**Depends on:** [T007 — Browse past identifications](T007.md)

## Goal

Let users tap a history entry and see the same full result view as a fresh identification — image preview plus the complete top-k list — reconstructed entirely from stored data, with **no** API call. Clearly mark the view as a saved/past result.

## Approach

Extract the result presentation that currently lives inline in `HomeScreen` into a reusable `ResultView` widget, then render it from both the Search flow (fresh) and a new `HistoryDetailScreen` (saved). The detail screen takes a `HistoryEntry` and builds `ResultView` from `entry.imageRef` + `entry.predictions` — it never touches `ApiClient`.

```
ui/lib/
  widgets/result_view.dart         # NEW — ImagePreview + PredictionList (+ saved badge)
  screens/home_screen.dart         # render results via ResultView (refactor, no behavior change)
  screens/history_detail_screen.dart  # NEW — builds ResultView from a stored entry
  screens/history_screen.dart      # tile tap → push HistoryDetailScreen
```

### Proposed files

| File | Purpose |
|------|---------|
| `ui/lib/widgets/result_view.dart` | Shared layout: `ImagePreview` (bytes/url) + `PredictionList`; optional `saved` flag for the badge |
| `ui/lib/screens/history_detail_screen.dart` | Scaffold hosting `ResultView` from a `HistoryEntry`; no API client |
| `ui/lib/screens/home_screen.dart` | Replace inline preview+list with `ResultView` (selectors preserved) |
| `ui/lib/screens/history_screen.dart` | `onTap` → `Navigator.push` to the detail screen |

### Reopen behavior (US-10)

- Tapping a `HistoryTile` (T007) pushes `HistoryDetailScreen` for that entry.
- The detail screen decodes `imageRef` (data URL → bytes for `ImagePreview`, or passes the remote URL) and renders the **full** `entry.predictions` through the existing `PredictionList` — identical to a fresh result (same top-prediction emphasis + rows, US-03/US-06 carryover).
- **No inference:** `HistoryDetailScreen` must not construct or call `ApiClient`. Reconstruct purely from the stored entry.
- **Saved indicator:** show a clear marker that this is a past result — e.g. an app-bar title like *"Saved result"* plus a small badge/subtitle with the original `createdAt` ("Identified 3 days ago"). Keep it flat and consistent with the palette.

### Refactor guardrails

Extracting `ResultView` from `HomeScreen` must not change the Search flow's behavior or selectors. Keep `Key('top-prediction')`, the `PredictionList` `Semantics('Prediction results')`, `Key('image-preview')`, etc. intact so the S001 demo still passes. This is a structural refactor, not a redesign.

### Selectors for automation (T010)

- Detail screen root / saved badge: `Key('saved-result')` (and a `Semantics` label like "Saved result").
- Ensure the reopened top match is assertable (existing `Best match` text from `PredictionList` carries over).

## Acceptance criteria

- [ ] Selecting a history entry shows the same full result view as a fresh identification — image preview + top-k list (US-10)
- [ ] Reopening a saved result makes **no** API call (verifiable: no request to `/predict*`) (US-10)
- [ ] The view clearly indicates it is a saved/past result, not a new run (US-10)
- [ ] `ResultView` is shared by Search and history detail; `HomeScreen` selectors/behavior unchanged; S001 demo still passes
- [ ] `flutter analyze` reports no new warnings

## Out of scope

- Re-running inference or "refresh prediction" on a saved entry
- Editing a saved entry's data
- Delete-from-detail and clear-all (T009) — though a delete affordance may be wired here if trivial, it's owned by T009
