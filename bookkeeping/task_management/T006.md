# T006 — Local history store + auto-save

**Status:** planned  
**Maps to:** US-08 (auto-save recent identifications)  
**Depends on:** [T005 — App shell + bottom tab navigation](done/T005.md)

## Goal

Persist every successful identification to browser-local storage automatically, with no extra "save" step. This task delivers the data model, a storage-backed `HistoryStore`, and the auto-save hook in the Search flow. Browsing/reopening/clearing the saved entries are built on top in T007–T009.

## Approach

Add a `HistoryEntry` model and a `HistoryStore` (`ChangeNotifier`) that serializes a list of entries to JSON and persists it in the browser via `shared_preferences` (which is backed by `localStorage` on web — satisfies "browser-local storage", local-only, no server). Expose a single `HistoryStore` instance to the widget tree so the Search flow can append and T007/T009 can read/mutate.

On a **successful** identification in `HomeScreen._identify`, build a `HistoryEntry` and call `historyStore.add(entry)`. Failures and empty/error states are not saved.

```
ui/lib/
  models/
    prediction.dart          # unchanged (reused)
    history_entry.dart       # NEW — id, createdAt, source, imageRef, predictions
  services/
    history_store.dart       # NEW — load/add/delete/clear over shared_preferences
  screens/home_screen.dart   # save on successful identify
  main.dart                  # construct + provide HistoryStore
```

### Proposed files

| File | Purpose |
|------|---------|
| `ui/lib/models/history_entry.dart` | Serializable entry; `toJson` / `fromJson`; holds `List<Prediction>` |
| `ui/lib/services/history_store.dart` | `ChangeNotifier` with `load()`, `add()`, `delete(id)`, `clear()`, `entries` (most-recent-first) |
| `ui/lib/screens/home_screen.dart` | Call `add()` after a successful `_identify` |
| `ui/lib/main.dart` | Create `HistoryStore`, `load()` on startup, pass to `AppShell` / `HomeScreen` |
| `ui/pubspec.yaml` | Add `shared_preferences` |

### `HistoryEntry` shape

```dart
class HistoryEntry {
  final String id;            // unique (e.g. timestamp + counter / UUID)
  final DateTime createdAt;   // when identified (US-08 timestamp)
  final HistorySource source; // upload | url
  final String imageRef;      // data URL (uploads) or remote URL (URL submissions)
  final List<Prediction> predictions; // full top-k, so reopen needs no API (US-10)
}
```

- **Image reference (US-08 "image or its reference"):**
  - **URL submissions** → store the original URL in `imageRef`.
  - **Uploads** → store a base64 **data URL**. To respect the ~5 MB `localStorage` budget, downscale the saved image (e.g. longest edge ≤ ~512 px, JPEG/WebP) before encoding; this is enough for the thumbnail (T007) and the reopened preview (T008). Keep the full top-k regardless.
- Serialize `predictions` using `Prediction`'s `class` + `confidence` (mirror `Prediction.fromJson`); the existing getters re-derive readable labels on load.
- `createdAt` stored as ISO-8601; render relative time in T007.

### Persistence details

- Single key (e.g. `driftid.history.v1`) holding a JSON array; bump the suffix if the shape changes.
- `load()` runs once at startup (tolerate missing/corrupt JSON → start empty, never throw into the UI).
- `add()` prepends (or stores then sorts desc by `createdAt`) and persists immediately, then `notifyListeners()`.
- Optionally cap retained entries (e.g. most recent ~50) to stay within storage limits — note the cap if applied.
- **Local-only:** no new network calls; nothing about history is sent to the API (US-08).

### Auto-save hook

In `_identify`, after `setState(() => _predictions = results)` on success, construct and add the entry. The save is fire-and-forget relative to the result view (don't block rendering results on persistence) and must not surface storage errors as inference errors.

## Acceptance criteria

- [ ] A successful identification is saved automatically — no extra "save" action (US-08)
- [ ] Each entry records the image (data URL) or its reference (URL), the predictions (incl. top), and a timestamp (US-08)
- [ ] History persists across page reloads and new browser sessions via browser-local storage (US-08)
- [ ] Nothing about history is uploaded to a server — storage is local only; no new API calls (US-08)
- [ ] Corrupt/absent stored data degrades gracefully to an empty history (no crash)
- [ ] `flutter analyze` reports no new warnings; existing demo (`npx playwright test`) still passes

## Out of scope
