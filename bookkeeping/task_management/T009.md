# T009 — Manage & clear history

**Status:** planned  
**Maps to:** US-11 (manage and clear history)  
**Depends on:** [T007 — Browse past identifications](done/T007.md)

## Goal

Give users control over what's stored on their device: delete a single history entry, or clear the whole history in one action (behind a confirmation). Deletions update browser-local storage immediately, and clearing returns the user to the empty-history state.

## Approach

Wire the History UI to `HistoryStore.delete(id)` and `HistoryStore.clear()` (implemented in T006). Add a per-entry delete affordance on the list and a "Clear all" action on the History screen that opens a confirmation dialog before clearing. Because the list listens to `HistoryStore`, removals re-render automatically; emptying the store shows the T007 empty state.

```
ui/lib/
  screens/history_screen.dart   # "Clear all" action + confirm dialog
  widgets/history_tile.dart     # per-entry delete affordance
```

### Proposed files

| File | Purpose |
|------|---------|
| `ui/lib/screens/history_screen.dart` | App-bar/overflow "Clear all" → `AlertDialog` confirm → `HistoryStore.clear()` |
| `ui/lib/widgets/history_tile.dart` | Delete control (icon button or `Dismissible`) → `HistoryStore.delete(entry.id)` |

### Delete a single entry (US-11)

- Each tile gets a delete affordance — a trailing `IconButton(Icons.delete_outline)` and/or a `Dismissible` swipe. Keep it flat and obvious.
- Calls `HistoryStore.delete(entry.id)`, which removes the entry from the in-memory list **and** persists immediately (US-11), then `notifyListeners()` so the row disappears.
- A brief `SnackBar` ("Removed from history") is fine; an undo is optional and out of scope if it complicates storage.

### Clear all (US-11)

- A "Clear all" action on the History screen (app-bar action or overflow menu), disabled/hidden when history is already empty.
- Tapping opens a confirmation `AlertDialog` (e.g. *"Clear all history? This removes every saved identification from this device."*) with Cancel / Clear.
- Confirming calls `HistoryStore.clear()`, which wipes the stored key immediately, after which the screen shows the empty-history state (T007).

### Selectors for automation (T010)

- `Key('clear-history')` on the clear action; `Key('confirm-clear')` on the dialog's confirm button.
- `Key('delete-entry-<id>')` on each tile's delete control.

## Acceptance criteria

- [ ] A single history entry can be deleted (US-11)
- [ ] The entire history can be cleared in one action, with a confirmation step (US-11)
- [ ] Deletions remove the data from browser-local storage immediately (survives reload) (US-11)
- [ ] After clearing, the empty-history state is shown (US-11)
- [ ] Controls are flat, labeled, and reachable by Playwright (`Key`/`Semantics`)
- [ ] `flutter analyze` reports no new warnings; S001 demo still passes

## Out of scope

- Multi-select / bulk-delete-by-checkbox
- Undo history / trash bin beyond an optional SnackBar undo
- Export before clearing, or per-entry editing
