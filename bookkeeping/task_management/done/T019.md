# T019 — Light-theme palette cleanup

**Status:** done  
**Maps to:** US-12 (color theme), US-06 (readable surfaces)  
**Depends on:** [T014 — Image selection card](T014.md) (so the card already reads `colorScheme`)

## Goal

Tidy up the **light** theme palette so the image selector reads as purple (the brand's primary pastel, `thistle`) instead of the current light-blue (`icyBlue`). Blue stays in the palette but is demoted to an accent/highlight role rather than the fill behind a large, prominent surface like the image card. Dark theme is unaffected.

## Approach

Today the light `ColorScheme` in `ui/lib/main.dart` maps the neutral/large surface roles to blue:

```26:35:ui/lib/main.dart
  static const Color icyBlue = Color(0xFFBDE0FE);
  static const Color skyBlue = Color(0xFFA2D2FF);
```

```91:115:ui/lib/main.dart
      secondary: DriftColors.skyBlue,
      secondaryContainer: DriftColors.icyBlue,
      ...
      surfaceContainerHighest: DriftColors.icyBlue,
```

The image selection card (T014) derives its fill from `Theme.of(context).colorScheme` (a container/surface role), so it currently picks up `icyBlue`. The fix is a **palette remap only** — no widget restructuring:

- Move the large neutral surface roles that the image card and similar broad fills resolve to onto the **purple** family (`thistle` / a soft purple container) instead of `icyBlue`.
- Keep blue (`skyBlue` / `icyBlue`) for **accent/highlight** roles only — small emphasis like the confidence chip in `ResultView` (`secondaryContainer`) or selection highlights — not as the background of a primary input surface.
- Verify nothing in the dark theme changes; only the light `ColorScheme` is touched.

### Proposed files

| File | Purpose |
|------|---------|
| `ui/lib/main.dart` | Remap light `ColorScheme` surface roles from `icyBlue` → purple family; keep blue as accent only. Add a soft-purple container constant to `DriftColors` if needed. |

### Notes

- Keep it **flat** per `AGENTS.md` — solid pastel fills, no gradients/shadows.
- Don't hardcode colors in widgets; the fix lives in the `ColorScheme` so every consumer (image card, surfaces) inherits it.
- Decide concretely which role the image card resolves to and confirm `thistle`/purple has enough contrast with `onSurface`/`ink` text and the card border.
- Reserve `secondary`/`secondaryContainer` (blue) for genuine highlights/accents so blue still appears, just not as the dominant selector fill.
