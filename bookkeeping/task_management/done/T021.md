# T021 — End-to-end product tour demo

**Status:** done  
**Maps to:** US-01 (upload), US-02 (image URL), US-03/US-14 (results + Result tab), US-07 (tab nav), US-09 (browse history), US-10 (reopen), US-11 (manage/clear), US-12 (theme)  
**Depends on:** [T017 — Demo recording + regression](done/T017.md), [T020 — Real car images in default test data](done/T020.md)

## Goal

Record a single, continuous Playwright clip that walks the whole DriftID product in one take — two identifications (an uploaded Toyota and a pasted Porsche URL), a tour of History (browse → reopen → back → delete one → clear all), and the theme switch (Light / Dark / System). This is a **new** end-to-end "tour" video for the PR/portfolio that sits alongside the existing focused record clips; nothing is removed and the fast headless `check` specs (the functional pass/fail gate) are untouched.

## Approach

Add one new recorded spec, `ui/demos/record-tour.spec.ts`, that scripts the full journey end-to-end against a real backend, holding frames between beats so the WebM reads as a guided walkthrough. Because the tour both *creates* its own History (two fresh identifications) and then needs a couple of extra entries to demonstrate delete/clear convincingly, seed a small History up front with `seedHistory(...)` so the list is already populated when the tour reaches it; the two live identifications then stack on top.

The existing record clips stay as-is — the tour is additive, giving us one whole-product walkthrough plus the existing focused single-feature clips:

```
ui/demos/
  record-tour.spec.ts            # NEW — the full guided tour (this task)
  record-demo.spec.ts            # KEEP — focused upload → Result clip
  record-history.spec.ts         # KEEP — focused save → reopen clip
  record-settings-theme.spec.ts  # KEEP — focused theme-switch clip
  history-*.spec.ts              # KEEP — headless `check` gate, unchanged
  history-reopen.spec.ts         # KEEP — no-API-call assertion stays the gate
```

Keep the existing recording plumbing: the `record` project in `playwright.config.ts` already matches `record-*.spec.ts`, sets `slowMo`, and writes a WebM at `test-results/**/video.webm`, so no config change is needed — the new spec is picked up automatically.

### Tour script (one test, in order)

1. **Setup** — `seedHistory(page, [...])` with two older entries (e.g. a BMW and an Audi, using `SEED_CAR_IMAGE` / `SEED_CAR_IMAGE_ALT`) so History isn't empty when we get there. `goto('/')`, enable semantics (`flt-semantics-placeholder` click).
2. **Search → upload (Toyota)** — click the image card (`Add a car photo`), `setFiles('demos/fixtures/toyota.jpg')`, press `Identify`; assert the app auto-switches to the **Result** tab and `Best match` is visible; hold the frame.
3. **Search → URL (Porsche)** — back to Search (now clean), fill the URL field (`Image URL`) with the served Porsche URL, watch it preview in the card, press `Identify`; assert Result tab shows `Best match` again; hold the frame.
4. **History — browse** — open the History tab; both fresh identifications plus the seeded entries are listed; scroll/hover the tiles so the populated list reads clearly on video.
5. **Reopen → back** — tap a tile; assert the shared Result view opens marked `Saved result` with `Best match`; switch back to History (the "back" beat).
6. **Delete one** — use a per-tile `Delete`; assert `Removed from history` and that the targeted tile is gone while others remain.
7. **Clear all** — `Clear all` → confirm `Clear`; assert the `No identifications yet` empty state.
8. **Settings — theme** — open Settings; click `Dark` (app darkens), then `Light`, then `System`, holding each frame so the re-theme is obvious on video.

### Fixtures

| File | Purpose |
|------|---------|
| `ui/demos/fixtures/toyota.jpg` | Real Toyota photo for the upload beat (replaces the generic `sample-car.jpg` for this clip) |
| `ui/web/porsche.jpg` | Real Porsche photo bundled into the static web build so the URL beat resolves offline |

**URL beat without external network:** the `/predict-url` backend downloads the URL server-side, so it must be reachable from the test host. Bundle the Porsche image under `ui/web/` (Flutter copies it into `build/web/`), and point the URL field at the local static server: `http://localhost:${process.env.WEB_PORT ?? '8080'}/porsche.jpg`. This keeps the tour deterministic and offline-safe (no third-party image host), matching the T020 rationale for History fixtures.

