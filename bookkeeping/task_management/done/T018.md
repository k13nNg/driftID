# T018 — Live URL image preview + shared preview component

**Status:** done  
**Maps to:** US-02 (submit an image URL), US-13 (focused Search screen), US-09 (browse history — carried)  
**Depends on:** [T015 — URL input + Search relayout](T015.md) (which depends on [T014](T014.md)); pairs with [T016 — Result tab + shared result surface](done/T016.md) for the History/Result reuse

## Goal

Show a live preview of a pasted/typed image URL inside the Search image card **before** the user presses `Identify`, instead of only after a successful identification. Consolidate the duplicated image-rendering logic (picked bytes vs. network URL, with loading + error placeholder) into a single shared preview component, and reuse it across the `ImageSelectionCard`, the History thumbnails, and the Result view so a URL-sourced image previews consistently on Search and shows up in the History tab.

## Approach

Today `HomeScreen` only sets `_previewUrl` inside `_identify` *after* a successful URL prediction (`home_screen.dart`), so pasting a URL displays nothing until inference returns. Drive the preview from the URL field instead: the existing `onChanged` (which already clears a picked file) also updates a live preview URL that feeds the card's `url`. Clearing the field (or picking a file) clears the preview, preserving the current mutual-clearing behavior. Previewing makes **no** API/inference call — it's a plain `Image.network`.

The network/memory rendering (bytes → `Image.memory`, url → `Image.network` with a loading indicator and a "Preview unavailable" placeholder, on the neutral grey surface) is currently duplicated in `ImagePreview`, the T014 `ImageSelectionCard`, and `HistoryTile._Thumbnail`. Extract it into one shared widget parameterized by `BoxFit` (card → `contain`, history thumbnail → `cover`) so all three render identically, and a URL load failure degrades to the same flat car placeholder everywhere.

History already stores a URL submission's remote address as `imageRef` and `HistoryTile._Thumbnail` already loads it via `Image.network`; this task routes that through the shared component and verifies URL-sourced entries actually render their image in the History list (and in the reopened Result via T016), with a graceful placeholder on failure.

```
ui/lib/
  widgets/image_preview_content.dart   # NEW (or refactor of image_preview.dart) — shared bytes/url renderer
  widgets/image_selection_card.dart     # T014 — live URL preview takes precedence over empty state
  screens/home_screen.dart              # URL onChanged sets a live _previewUrl (not only post-identify)
  widgets/history_tile.dart             # thumbnail renders via the shared component
```

### Proposed files

| File | Purpose |
|------|---------|
| `ui/lib/widgets/image_preview_content.dart` | Shared `({ Uint8List? bytes, String? url, BoxFit fit })` renderer: `Image.memory` / `Image.network` with loading + "Preview unavailable" placeholder; single source of truth for the neutral surface treatment |
| `ui/lib/widgets/image_selection_card.dart` | Accept a live `url`; when set (and no bytes), preview the remote image in place — tap still opens the picker, expand-to-inspect still works |
| `ui/lib/screens/home_screen.dart` | URL `onChanged` updates a live preview URL feeding the card; clear it when the field empties or a file is picked |
| `ui/lib/widgets/history_tile.dart` | Render `_Thumbnail` via `image_preview_content.dart` instead of bespoke `Image.network`/`Image.memory` |

### Preview behavior (US-02 / US-13)

- **Live preview:** a non-empty URL in the field previews immediately in the card (loading indicator while it fetches). No inference/API call is triggered just to preview.
- **Graceful failure:** an invalid/unreachable URL shows the existing "Preview unavailable" placeholder — never an error banner or crash.
- **Tap-to-pick preserved:** a URL preview fills the card visually but tapping it still opens the file picker; picking a file clears the URL preview (and vice versa), matching the current mutual-clearing logic.
- **Expand-to-inspect:** the T014 expand affordance (`Key('expand-image')`) works for URL previews too.
- **History (US-09 / US-10):** a URL identification's image appears in the History tab thumbnail and in the reopened Result, the same as an upload — both go through the shared component.

### Selectors for automation (T017)

- Reuse `Key('image-card')` (T014) and the existing history-tile keys. Keep `Key('image-preview')` available on the shared component for assertions if `ImagePreview` is the thing being refactored.

- [x] Typing/pasting a URL shows a live image preview in the Search image card **before** `Identify`, with no inference call (US-02/US-13)

- [x] Bytes/network rendering (image + loading + placeholder) lives in one shared component reused by the card, History thumbnails, and Result view — no duplicated network-image logic
- [x] A URL identification's image shows in the History tab thumbnail and in the reopened Result (US-09/US-10)

- [x] `flutter analyze` reports no new warnings

## Out of scope

- URL validation/normalization beyond a basic non-empty check; CORS/proxy handling for remote images
- Caching previews, prefetching, or drag-and-drop upload
- Changes to inference, the `Prediction` model, or auto-save semantics
- Playwright updates for the new preview flow (T017)
